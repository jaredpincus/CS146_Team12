Web Programming

Internet Protocols

Link Layer

  • Responsible for the physical transmission of data across media.
  • Handles packet creation, transmission, reception, error detection, and collisions.

Internet Layer

  • Provides a the best way of connection.
  • Makes use of IP addresses.

Transport Layer

  • Ensures transmissions arrive in order and without error.

Application Layer

  • Protocols that allow applications to interact with Transport Layer.
  • Some protocols include:
    • HTTP: web communication
    • SSH: secure shell connect to server
    • FTP: File transferring
    • POP/SMTP: Email transfer protocols

DNS and URL

Hostname System

  • Developed in the early days of TCP/IP.
  • Hostnames are alphanumeric names assigned to each computer in the system.
  • If an operating system encounters a hostname it consults a host file to find the respective IP address.
  • Works well on smaller systems, but becomes inefficient on larger networks.
  • Multiple computers cannot access the hostfile at the same time.

Domain Name System (DNS) and URL

  • DNS divides a namespace into hierarchical entities called domains.
  • A fully qualified domain name (FQDN) includes the domain name with the hostname.
  • URL stands for Uniform Resource Locators
  • Example: http://www.reddit.com/index.php?page=17#article
    • "http://" : protocol
    • "www" : 3rd level
    • "reddit" : 2nd level
    • "com" : top-level domain
    • "/index.php" : path
    • "?page=17" : query string
    • "#article" : fragment

HTTP and Browsers

  • HTML: hypertext markup language
  • HTTP: hypertext transfer protocol
  • HTTPS uses TLS to secure connection.
  • Both HTML and HTTP were developed at the same time by the same person.
  • Connectionless protocols disconnect from the server after making a request and when the response is ready the connection is re-established.
  • In a stateless protocol the client and server know about each other only during the request. Once the connection is closed they must reconnect as if it was the first time.
  • A request header includes data about client. A response header includes data of server.
  • Client request codes:
    • GET : retrieves the representation of pages
    • HEAD : same as GET, except without content
    • POST : delivers information to server

Project Management

Projects

What is a project?

  • A project is unique, temporary, requires resources, and has uncertainty and stakeholders.
  • Most effort is put forth during the middle of the projects life.
  • Most uncertainty exists at the end of the project.

Project Lifecycle

  • The most basic model for a project is the waterfall model.
  • In the waterfall model you only work forwards and do not go backwards in stages.
  • In CS the waterfall model is typically represented by:
    • Inception
    • Requirements
    • Analysis
    • Design
    • Implementation
    • Testing/Debugging
    • Production
    • Distribution
    • Maintenance
  • The Project Manager is responsible for organizing these steps properly and planning the schedule.

Project Managers

The Vital Dozen

  • Understand the context of project management.
  • Recognize project team conflict as progress.
  • Understand who the stakeholders are and what they want.
  • Accept and use the political nature of organizations.
  • Lead from the front.
  • Understand what "success" means.
  • Build and maintain a cohesive team.
  • Enthusiasm and despair are both infectious.
  • One look forward is worth two looks back.
  • Remember what you are trying to do.
  • Use time carefully or it will use you.
  • Above all, plan, plan, plan.

The Waterfall Model

Phases

  • Requirement Gathering and Analysis:
    • All the requirements of the project are captured in this phase.
    • The requirements are documented in a requirement specification doc.
    • The requirements doc must address these four key points:
      • - Discription of the problem
      • - Basic approach of solution
      • - Constraints (time, budget, etc)
      • - How success will be measured
  • System Design
    • The requirements from the previous phase are studied in this phase and designing begins.
    • Helps to specify hardware that may need to be required and helps define the overall architecture.
  • Implementation
    • Using inputs from the system design, small programs called units are developed.
    • Each unit is tested for functionality and are integrated into the next phase.
  • Integration and Testing
    • All the units from the previous phase are integrated into a system.
    • The created system is then tested for faults and errors.
  • Deployment of System
    • Once all testing and debugging is completed, the system can then be deployed to customers or the market.
  • Maintenance
    • Issues may arrise once the system is deployed to the customer environment.
    • This phase releases patches and updates to these issues.

Software Engineering

System Design

Object Oriented Design

  • Encapsulation
    • The grouping of related ideas into one unit and then can be referenced by a single name.
    • Encapsulation allows abstracting which hides things away from the user when they do not need to know how it operates.
  • Inheritance
    • Inheritance occurs when subclasses are derived from parent classes and act as refinements of the parent.
    • This is done so code does not become repetitive.
  • Polymorphism
    • The ability for an object to take on many forms.
    • Most common use is when a parent class references a child class object.

Testing

Testing vs Debugging

  • Testing is the means of determining if a bug exists.
  • Debugging is the process of trying to find the bug in the code.

Types of Errors

  • Syntax Errors
    • Occurs when a typo exists.
    • Easiest to find since they are picked up during compilation.
  • Run-time Errors
    • Program crashes during execution.
    • An example of this is a divides by zero error.
  • Logic Errors
    • Hardest errors to pick up.
    • Program does not crash, but instead gives the wrong output.
    • Results from incorrect thought processes.

Types of Testing

  • Unit Testing: Also known as component testing, verifies functionality of a secific section of code.
  • Integration Testing: Seeks to verify the interfaces betwen components (modules) against a software design.
  • Component Interface Testing: Checks handling of data between system components.
  • System Testing: Tests a completely integrated system.
  • Acceptance Testing: Sponsor validates the system is up to par and is what was asked.

Testing approaches

  • Black-box testing
    • - Testing without knowledge on how the method works
    • - Just tests inputs/outputs
  • White-box testing
    • Tests every line of code in program
    • Checks internal workings of an application as opposed to functionality

HTML

Structure

  • <html> contains all other HTML elements in the document
  • <head> contains descriptive elements about the document
  • <body> contains content to be displayed by the browser
  • <meta> declares the character encoding is UTF-8
  • <link> specifies external css file
  • <script> specifies external JavaScript file

Elements

  • <p> container for text and other HTML elements
  • <div> container element used for logical grouping of content
  • <a> anchor element used when adding links to a page

Special Character Entities

  • Can get special symbols by typing "&" followed by one of the following strings:
  • nbsp; nonbreakable space  
  • lt; less than symbol <
  • gt; greater than symbol >
  • copy; copyright symbol ©
  • trade; trademark symbol ™

List Elements

  • <ul> unordered list
  • <ol> ordered list
  • <dl> description list
  • <li> list item

Tables

  • <table> defines a table
  • <tr> defines a table row
  • <th> defines a table header
  • <td> defines table data/cell
  • <caption> adds a caption to a table
  • To make a cell span more than one column use the attribute colspan
  • To make a cell span more than one row use the attribute rowspan

Semantic Structure Elements

  • <header> contains the documents heading
  • <main> contains the main unique content of the document
  • <section> broad element for larger sections of content
  • <article> used for blocks of content that can be read
  • <aside> used for sidebars or grouping of nonessential elements like ads

CSS

CSS Styles

Inline styles

  • Style rules placed within an HTML element via the style attribute.
  • Example: <h2 style="font-size: 24pt;"> Description </h2>

Embedded style sheet

  • Style rules placed in a style tag inside the head tag.
  • Example: <style> h2 { font-size: 24pt; } </style>

External style sheet

  • Style rules placed within and external text file with the .css extension.
  • External style sheets are implemented using the link tag.
  • Example: <link rel="stylesheet" href="styles.css"/>

Selectors

Element Selectors

  • Select all instances of a give HTML element.
  • Example: h2 { font-size: 24pt; }

Class Selector

  • Allows you to simultaneously target different HTML elements regardless of their position in the document.
  • Example:
  • <style> .first { font-size: 24pt; } </style>
  • <p class="first">Reviews</p>

Id Selector

  • Allows you to target a specific element by an id attribute regardless of type or position.
  • Example:
  • <style> #first { font-size: 24pt; } </style>
  • <p id="first">Reviews</p>

Attribute Selector

  • Provides a way to select HTML elements either by the presence of an element attribute or value of an attribute.
  • Example:
  • <style> [title] { font-size: 24pt; } </style>    This will target all elements with a 'title' attribute.

Pseudo Selectors

  • Pseudo-element selector: Select something that does not exist explicitly as an element in the HTML document by still a recognizeable object.
  • Pseudo-class selector: Applies to an HTML element, but targets either a particular state, or a variety of family relationships.
  • Pseudo-element and Pseudo-class selectors:
    • a:link
    • a:visited
    • :focus
    • :hover
    • :active
    • :checked
    • ::first-child
    • ::first-letter
    • ::first-line
